# Normal Long Task

## Boundary

This package-managed Skill is generated by `ty-context sync`. It is an acceptance-preparation tool, not an execution framework or completion authority. It inherits Context Priority and source-to-Context judgment while preparing acceptance, and replaces the former ordinary long-task plan/matrix/verdict artifact set with only the bounded outputs below. It does not replace or execute the default implementation workflow.

Invoke only through:

```text
/normal-long-task
```

Do not trigger from task length, file count, broad plan/acceptance words or inferred complexity. Do not execute implementation, mutate durable Context, run side-effect validation or claim any item complete unless the user separately asks for a current evidence audit.

For strict machine completion use explicit Composite only: `/prepare-composite-long-task` for a raw multi-SFC campaign or `/composite-long-task-workflow` for one already-complete three-input SFC.

## Outputs

A completed invocation produces only:

1. a preserved UTF-8 source copy of every user source under `tmp/ty-context/plan-acceptance/**`;
2. one complete acceptance checklist;
3. a compact goal/target prompt only when requested;
4. a Local Audit only when the user explicitly requests persistent recovery/progress state or the requested target mode requires resumability.

Do not create a Plan Conformance Matrix, Final Acceptance Verdict, separate test-requirements file, evidence ledger file, second execution plan or default Local Audit.

The source is implementation direction, not proof. The checklist is the complete acceptance standard, not proof. An optional prompt is a navigation wrapper, not authority. An optional Local Audit records progress only and cannot mark completion.

## Input Authority

Read in this order:

1. the current user instruction;
2. every referenced/pasted source;
3. `project_context/global.md`, `project_context/architecture.md`, `project_context/context.toml` and graph-relevant Context;
4. repository guidance and relevant local Skills;
5. code/tests only to discover real surfaces, paths, commands and evidence opportunities.

Never lower the source target to current implementation convenience. If source and durable Context conflict on product scope, ownership, architecture, API/schema, state/recovery, security or acceptance meaning, preserve the source, report the exact conflict and ask for the genuine decision before producing a final checklist.

## Source Preservation

- Create `tmp/ty-context/plan-acceptance/` when needed.
- Copy external files byte-for-byte in meaning and preserve pasted text verbatim. Do not summarize, normalize, reorder, translate or rewrite it.
- Use stable readable `<slug>.md` names.
- A source already under the directory may be reused.
- Stop if a referenced source cannot be read.
- For a two-document packet, preserve `Development Plan / 开发方案` and `Acceptance and Tests / 验收清单和测试用例` separately.

Recommended paths:

```text
tmp/ty-context/plan-acceptance/<slug>.md
tmp/ty-context/plan-acceptance/<slug>-acceptance-checklist.md
tmp/ty-context/plan-acceptance/<slug>-local-audit.md   # opt-in only
```

## Two-Document Strict Mode

When both a Development Plan and Acceptance-and-Tests source are supplied, treat them as one strict upstream packet. It must expose:

- objective/original requirement scope;
- implementation direction and boundaries;
- acceptance IDs/items;
- required evidence and verification method;
- failure/invalid-evidence rules;
- required automated tests or an explicit test scope;
- real user/runtime paths or explicit non-UI/non-runtime scope;
- completion-state rules;
- blockers or an explicit no-blocker statement.

If any required field is missing, preserve both sources, list missing fields and stop. Do not infer a replacement checklist or prompt.

## Checklist Reuse

If the source has an explicit concrete acceptance checklist:

- copy every checklist section in source order into the separate checklist file;
- preserve wording, language, order, IDs, checkboxes/tables, evidence, test and blocker notes verbatim;
- do not strengthen, translate, normalize, merge, split or add items;
- keep the source and checklist as separate files;
- use the full checklist path from the prompt when it is too large to inline.

For a two-document packet, reuse the Acceptance-and-Tests source only when it already has IDs, scope, evidence, verification, failure, completion-state and invalid-evidence rules. Otherwise strict mode stops.

If no explicit checklist exists, generate one from the full source and relevant Context.

## Generated Checklist Contract

Every material source section maps to an acceptance item or an explicit non-goal. Convert broad claims such as all, full, maximum, complete, formal, production-ready or verified into a bounded inventory and observable evidence.

Each generated item includes:

- stable AC ID;
- source reference and scope;
- expected observable state;
- required evidence layer(s);
- verification method/command or path;
- failure condition;
- invalid or insufficient evidence;
- initial state `unknown / not_run`.

Use only applicable evidence layers: code, API/schema/data, runtime/state/recovery, user-visible UI/browser path, security/redaction, generated artifact, automated test/build/typecheck/lint/smoke, external CI or human acceptance.

Test requirements live inside the checklist. For each required automated test record behavior/path, covered AC, command, expected failing condition before implementation when applicable, expected passing condition after implementation, failure condition and whether it is required or auxiliary. Never create a fourth test-requirements artifact.

UI-facing acceptance requires the real page/route and matching user-visible state unless explicitly scoped out; component/viewmodel/mock/unit evidence alone is auxiliary. Runtime/API/schema acceptance requires its real implementation and observation surface. Old, stale, mismatched, unexercised, code-only or partial evidence cannot prove current completion.

Hard blockers remain non-completion. Record what is blocked, local attempts, missing evidence, acceptance impact, minimum user/external action and any explicitly authorized fallback.

## Optional Local Audit

Create a Local Audit only when explicitly requested for recovery/progress persistence or when the explicitly requested target mode requires resumability. It records:

- current overall/AC progress;
- commands run and current results;
- failure reasons and invalid/stale evidence;
- remaining local work;
- external/decision blockers.

It starts from `unknown / not_run`. Fresh contradictions downgrade the affected item. It is not Context, a second plan, a matrix, a verdict, an evidence authority or product proof, and no default Workflow or Composite completion state reads it.

## Optional Target Prompt

When requested, emit a paste-ready prompt no longer than 3850 characters including line breaks.

- First line identifies the source path and says it is source/implementation direction, not proof.
- Second line identifies the full checklist and says it is the complete acceptance standard.
- Mention the Local Audit path only when the user requested one; otherwise do not invent it.
- Require maximum safe autonomous progress under current instructions, project Context and permission boundaries.
- Require fresh project evidence and continued work while any checklist item is unmet.
- Require real UI/runtime/API paths where applicable and reject stale/partial/contradictory evidence.
- Require minimal user action only after safe local discovery/access/elevation attempts show a genuine credential, MFA, permission, approval, payment, legal or semantic blocker.
- Preserve source/checklist authority and do not introduce another execution framework, plan, matrix or verdict.

Chinese prompts that include a checklist must say: `该文件是完整验收标准，验收以这个为准。完成前必须逐项检查，不满足则继续实现。`

## Final Response

Report:

- preserved source path(s);
- full checklist path and whether it was reused verbatim or generated;
- optional prompt in a code block when requested;
- optional Local Audit path only when created;
- unresolved source/Context ambiguity or blocker.

Do not claim acceptance passed without a separately requested current evidence audit.

## Forbidden Behaviors

- Do not execute the source or generated prompt.
- Do not create matrix/verdict/evidence-ledger/second-plan/default-audit artifacts.
- Do not register any artifact in `project_context/context.toml`.
- Do not hide missing source fields, conflicts or blockers.
- Do not rewrite an explicit source checklist.
- Do not reduce full requested scope using current code, a sample, stale result or shorter prompt budget.
- Do not hardcode business-specific providers, APIs, pages, schemas, statuses or commands in this package-managed Skill.
