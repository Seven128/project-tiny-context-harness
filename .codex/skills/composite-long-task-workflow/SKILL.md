# Composite Long-Task Workflow V2

This package-managed Skill is generated by `ty-context sync`. Consumer projects must not edit it directly.

## Invocation boundary

Run only through explicit `/composite-long-task-workflow` invocation. Ordinary questions, bug fixes and long tasks do not start this workflow or create a workdir. Agent planning, decomposition, native subagents, implementation technique, tests and reviews remain implementation choices outside the workflow contract.

## Purpose

The workflow permits implementation drift while preventing drifted delivery. Completion meaning comes only from the frozen three-input contract; facts come only from the active verifier; final acceptance comes only from one fresh all-spec final snapshot; the trusted Stop Hook decides whether Codex may end.

Every required step must reduce a named drift risk:

- Contract compilation prevents omitted requirements, uncovered obligations/ACs and unverifiable acceptance.
- Intermediate active verification finds behavior, scope and evidence drift before final delivery.
- Final full verification prevents stale, partial or cross-snapshot proof.
- The Stop Hook prevents an unaccepted task from being reported as complete.

## Required inputs

The workdir must contain complete strict YAML authorities:

- `product-architecture-source.yaml` (`product-source-v2`)
- `technical-realization-plan.yaml` (`technical-plan-v2`)
- `acceptance-checklist.yaml` (`acceptance-checklist-v2`)

The graph must cover Product Requirement → atomic PI obligation → AC → executable frozen verification spec. Product boundaries, non-completing outcomes and forbidden shortcuts require executable negative assertions. Owner UI surfaces require browser verification. Full-population claims require a frozen enumerator and 100% coverage rule. Manual-only, summary-only and verifier-free ACs are invalid.

Do not infer, repair or weaken missing contract meaning. Return compiler diagnostics to the user when a required product decision is absent. Product tasks cannot modify the oracle, verifier or Hook that proves the same task.

## Fixed loop

```text
ty-context composite-long-task init <workdir>
author the three strict YAML inputs
ty-context composite-long-task compile <workdir>
implement freely
ty-context composite-long-task verify <workdir>
while current-status is needs_work: repair findings and verify again
ty-context composite-long-task final-gate <workdir>
while final result is needs_work: repair and return to verification
accepted: report completion
externally_blocked: report only the blocker and minimum user action
```

`verify` executes only frozen executable/argv/cwd values, collects only verifier-owned fresh artifacts and writes `current-status.json`. The agent cannot provide commands, artifacts, evidence records, assertion results or AC/PI status. Intermediate results never authorize acceptance.

`final-gate` rechecks all source, Context, oracle, verifier and spec hashes; creates one new isolated source snapshot; runs every in-scope spec; recomputes every PR/obligation/AC; checks negative, scope and population invariants; and writes `final-result.json`. Historical attempts are never stitched together.

## Legal states

- `accepted`: the latest complete final snapshot passed and the current workspace still matches it.
- `needs_work`: internal loop state. It is never a legal final reply.
- `externally_blocked`: only fresh evidence of MFA, unavailable credentials/permission, required user contract decision, external approval, platform/legal restriction or persistently unavailable irreplaceable external service.

Test failures, missing behavior/evidence, UI gaps, scope drift and ordinary implementation problems are `needs_work`.

## Recovery and completion

Recovery reads only `compiled-contract.json`, `current-status.json` and the latest run result. `goal-objective.txt` is a thin recovery pointer, not an authority. Runtime results remain under `tmp/ty-context/plan-acceptance/**` and never enter `project_context/**`.

If the repository Hook is missing, disabled, untrusted, modified, conflicting or fails its smoke heartbeat, do not start: report `host_completion_gate_unavailable`. With an active task, SessionStart/PostCompact inject current identity/findings and Stop blocks any exit unless the latest final result is accepted and the workspace hash is unchanged. An externally blocked reply must not contain completion language.
