# Composite Long-Task Contract V3

This package-managed Skill is generated by `ty-context sync`. Consumer projects must not edit it directly.

## Invocation boundary

Run only through explicit `/composite-long-task-workflow` invocation. Ordinary questions, bug fixes and long tasks do not start this workflow or create a workdir. Agent planning, decomposition, native subagents, implementation technique, tests and reviews remain implementation choices outside the completion contract.

The workflow accepts only an already-complete three-authority bundle. Use `/prepare-composite-long-task` first when the input is a raw requirement that still needs Scope Fit, stable SFC selection or structured authoring.

## Completion authority

Implementation may drift; completion authority may not. The administrator-installed Managed Host Gate seals the first complete Contract V3 in the workspace-external Host registry, outside repository write access. The repository/workdir copies are signed UX mirrors, not authority. The Harness owns assertions, bindings, population coverage, counterfactual sensitivity and every bottom-up entity result; an Oracle returns typed actual observations only.

- `compile` freezes the full Requirement → PI → Obligation → Binding → AC → Proof → Spec → Counterfactual graph, registered Context, Oracle bundles and transitive inputs, command targets, dependency/sandbox identities and managed policy.
- `verify` runs affected frozen specs and returns verifier-owned `needs_work` findings; it never accepts.
- `final-gate` uses one final source snapshot and a fixed all-spec/probe/binding/counterfactual/entity recomputation before the Host durably signs a result.
- The managed-only Codex Stop Hook repeats the full final verification and permits completion only for the current committed accepted result.

## Required inputs

The workdir contains exactly the complete strict YAML authorities:

- `product-architecture-source.yaml` (`product-source-v3`)
- `technical-realization-plan.yaml` (`technical-plan-v3`)
- `acceptance-checklist.yaml` (`acceptance-checklist-v3`)

The exhaustive bidirectional graph must cover each Product Requirement, atomic PI obligation, observable implementation binding, AC, proof requirement, executable verification spec and obligation-specific counterfactual. Boundaries, non-completing outcomes and forbidden shortcuts require related executable negative assertions. Owner UI surfaces require related browser proof and a current trace. Full-population claims require a frozen enumerator, exclusions and 100% coverage. Manual-only, summary-only, verifier-free, unobservable or unrelated proof is invalid.

Do not infer, repair or weaken missing product meaning. Return compile diagnostics when a material product decision is absent. V2 sources, Observation V1, aliases, merges, tags, multi-document YAML, legacy workdirs and compatibility import are rejected.

## Fixed loop

```text
ty-context composite-long-task init <workdir>
place the three strict V3 YAML inputs
ty-context composite-long-task compile <workdir>
implement freely
ty-context composite-long-task verify <workdir>
while current status is needs_work: repair findings and verify again
ty-context composite-long-task final-gate <workdir>
while final result is needs_work: repair and return to verification
accepted: report completion
externally_blocked: report only the blocker and minimum user action
```

The public surface contains only `init`, `compile`, `verify`, `status`, `final-gate`, `stop-check` and `render-goal`. There is no force/reset/cancel, evidence registration, arbitrary verifier, slice/epoch/state/derived command or compatibility alias.

Real package-manager dependencies and Playwright browsers come only from verified content-addressed read-only layers. Node's Permission Model supplements the OS sandbox. Environment YAML names references only; Host secret providers inject declared values after snapshot and retained output/artifacts are redacted or destroyed on leakage.

## Legal states

- `accepted`: the latest complete Host-committed final recomputation passed and the current workspace/authority still matches it.
- `needs_work`: internal repair state; never a legal final reply.
- `externally_blocked`: only current trusted probes establish MFA, unavailable credentials/permission, a required user contract decision, external approval, platform/legal restriction or a persistently unavailable irreplaceable external service.

Test failures, missing behavior/evidence, UI gaps, scope drift, sandbox failures and ordinary implementation problems are `needs_work`. Oracle prose or an agent claim cannot create a blocker.

## Recovery and completion

Recovery reads the Host registry through the public CLI and uses `compiled-contract.json`, `current-status.json`, `final-result.json` and `goal-objective.txt` only as signed/thin views. Runtime results remain under `tmp/ty-context/plan-acceptance/**`; Host registry, keys, bundles and layers remain outside the workspace; neither enters `project_context/**`.

Do not start when system `requirements.toml`, signed release identities, privileged Host service, managed Hook, Codex-origin heartbeat or create-only registry is unavailable. A repository/user/plugin Hook is never a fallback. With no active task, all managed Hook events are a complete no-op. With an active task, SessionStart/PostCompact restore current findings and Stop blocks every stale, malformed, forged, historical, uncommitted, drifted, `needs_work` or invalid blocker reply.
