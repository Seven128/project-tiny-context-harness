# Project / Delivery Context

## Project Goal

- Maintain `project-tiny-context-harness`, the npm package and `ty-context` CLI behind the public Project Tiny Context Harness display name.
- Current product direction is Minimal Context Harness: preserve the smallest durable facts and workflow contract needed for fresh-agent recovery, iteration, debug and requirement changes.
- Design purpose: Tiny Context has two core layers. Minimal Context defines the durable fact layer; Workflow Contract defines the agent behavior layer for reading Context, deciding `Context Delta`, compiling Task Contracts, using temporary `plan.md` / equivalent plan surfaces when needed, implementing against Context and checking conformance. Together they keep project intent, implementation and verification evidence aligned without restoring stage artifacts.
- Design problem: when downstream work exposes a missing upstream capability, current code can show a convenient edit path but cannot decide whether changing upstream modules is allowed project intent. The Harness preserves that repo-owned intent layer in `project_context/**`.
- Current best practice: short tasks use the workflow contract plus Context layer directly. Long-running ordinary acceptance preparation still uses `/normal-long-task` when useful. Superpowers-backed long-running execution is now a three-input packet plus state-backed workflow: Product / Architecture Source owns intent and scope, Technical Realization Plan owns the executable blueprint, and Acceptance Checklist owns completion authority. When those inputs exist and the next step is Superpowers-backed execution, invoke `/composite-long-task-workflow`; it produces a thin Codex Goal objective plus frozen `workflow-protocol.md`, `execution-binding.md` and canonical state entry under `tmp/ty-context/plan-acceptance/**`, then the executor uses the specific Superpowers plugin/workflow to derive concrete implementation slices through the workflow contract plus Context layer.

## Non-goals / Boundaries

- The package does not replace project tests, CI, review or human acceptance.
- New consumer projects should not default to lifecycle phases, stage task state, stage skills, stage work-product trees or phase gates.
- `sync` refreshes managed assets and default Context authoring Skills; legacy stage semantic migration has been removed, while safe Schema v4 upgrade migrations remain.
- Legacy stage assets are not shipped as a runnable default or compatibility layer.

## Background

- This repository contains the Harness package source, npm package release/source-sync logic and delivery benchmark logic.
- Earlier versions used a stage-based Tiny Context workflow with lifecycle state, plan tasks, stage skills, stage work-product trees and many validators.
- Benchmark pilots showed that full default ceremonies and frequent workflow gates create substantial time/token friction on ordinary and medium-complexity tasks.

## Design Rationale

- The durable value with the clearest expected return is context recovery and priority discipline, not forcing a full Tiny Context document chain for every project.
- The historical stage-based Harness externalized the whole Tiny Context through lifecycle state, plan tasks, PRD / tech plan / implementation / review / test / release artifacts and phase gates. Benchmark pilots showed those writes, transitions and gates are objective time/token cost; details are summarized in `PROJECT_SPEC.md`.
- Modern coding agents have internalized much of the ordinary single-task loop: compact requirement understanding, local design choice, code editing, test execution and simple repair. The default Harness should not duplicate that capability with broad ceremonies.
- ADR-level rationale is downgraded into `project_context/global.md#Design Rationale` or area Context when it still affects future work.
- Implementation facts should live in code, tests, comments and short area Context constraints when the code is not self-explanatory.
- Product/UIUX/development engineer Skill customization uses separate project-local Skills such as `<harnessRoot>/skills/product_plan/SKILL.md`, `<harnessRoot>/skills/uiux_design/SKILL.md` and `<harnessRoot>/skills/development_engineer/SKILL.md`; `sync` overwrites package-managed default `context_*` Skills and leaves separate local Skills untouched. Project-local Skill front matter `description` trigger keywords should stay aligned with the matching default Skill and the project `AGENTS.md` role-trigger rule.
- Architecture Context is intentionally restrained: it records durable boundaries, component relationships and constraints, not implementation narration.
- Schema v4 makes the lightweight Context graph the default: `project_context/context.toml` declares product/domain-owned areas, role-based context files, read triggers and monorepo boundary metadata while ordinary projects keep one default `main` area.
- `project_context/**` is authoritative for intended ownership, responsibility, architecture boundaries, integration direction, allowed/forbidden dependencies and verification/deployment entry paths; code is authoritative for current implementation state. Gaps between them should be treated as implementation drift, missing work or stale Context that must be called out.
- Context-first is the default workflow habit for changes to durable product or technical facts; before the first code edit, agents classify whether the task changes product ownership/plans, module responsibilities, information architecture, API/Schema, state or scheduler semantics, cross-domain boundaries or verification/deployment entry paths.
- `PROJECT_SPEC.md` owns the stable Harness mental model and core term vocabulary: Minimal Context Harness is a set of expected agent behavior constraints, not a document workflow.
- Context Priority Ladder is expected agent behavior: read Context first, let foundation / contract / decision-rationale / architecture / verification / deployment Context interpret current-code convenience, run the page product-positioning check for Web/page/layout/module-boundary/information-placement tasks, classify durable-fact impact or use `Context Delta` inside task-contract scenarios, choose context-first or code-first, then perform Contract Conformance when applicable and Context drift check before handoff.
- For Web page, frontend layout, UI/UX, product module boundary and information-placement tasks, the lightweight page product-positioning check runs before context-first classification and supplies the evidence for it; the check does not itself require a Context update unless it reveals durable page responsibility, information architecture, persistent-information boundary or ownership facts.
- Broad product/UIUX principles remain as judgment philosophy, while reusable control-task questions for inputs, selection, search, configuration, scheduling, budgets/quotas/limits and feedback states live in the default Skills; project-specific answers belong in Context or project-local Skills, and per-change Context Conformance evidence belongs in handoff/final/PR text.
- Long-running task execution should use package-managed temporary pre-execution surfaces because Context priority and drift checks are intentionally soft constraints. For short tasks, workflow contract plus Context layer usually keeps behavior aligned. For long tasks, Context itself can still capture the expected facts, but the Context-to-code step drifts under soft constraints as the context window, handoff count and verification loops grow. The ordinary long-task Skill (`/normal-long-task`) copies the referenced plan into `tmp/ty-context/plan-acceptance/**`, reuses an explicit concrete checklist already present in the plan or otherwise combines the plan with relevant Context to define falsifiable completion criteria, and may output a generic paste-ready target prompt. The Composite Long-Task Workflow Skill (`/composite-long-task-workflow`, Chinese display name `多组合长程任务工作流 Skill`) consumes Product / Architecture Source, Technical Realization Plan and Acceptance Checklist when they already exist, and renders a state-backed Superpowers workflow entry instead of treating target-mode text as the whole product. Its expected runtime effect is explicit fusion rather than agent improvisation: Tiny Context Workflow Contract first, then three-input source authority, `task-state.json` compilation, Superpowers implementation slices, canonical state/evidence updates, generated audit/matrix/verdict/progress/evidence views, slice/epoch gates, and final-gate completion. It freezes the package-managed detailed workflow into `workflow-protocol.md`, writes task-local `execution-binding.md`, and renders `goal-objective.txt` as a thin Codex Goal objective that points to the protocol and binding while retaining the authority model, state source, completion gate and blocker protocol. If only a product/architecture source plus checklist exists, or if the Technical Realization Plan is not executable enough to pass required field checks, the Skill stops with a Missing Fields Report instead of deriving one or generating execution entry artifacts. The Superpowers-backed path uses a canonical state kernel under `tmp/ty-context/plan-acceptance/**`: `task-state.json` is the only execution state source, `events.ndjson` is append-only, and `derived/**` audit/matrix/verdict/progress/evidence/context/final views are generated only. Slice deltas and canonical evidence records replace hand-synchronized local audit, evidence manifest, plan-conformance matrix, final verdict and progress ledger state. Generated Goal objectives and bindings treat existing local application/browser sessions, CLI auth and OS credential helpers as user-authorized self-service resources: executors should open the relevant app, page or tool and try the existing session before reporting login or access as blocked; only absent or failed login, MFA, missing permission or external approval becomes a user action. The generated Goal objective requires Product Context Delta and Technical Context Delta checks before implementation, then state-backed source coverage, Context binding, plan conformance, acceptance evidence and final completion gates. The Skill is a Tiny Context-owned composite workflow adapter for Superpowers-backed long-task execution, aligned to the official Superpowers skills rather than an upstream-owned schema; it is not the Tiny Context Workflow Contract itself, not a business fact source, not a generic target-mode prompt generator and not a Superpowers fork. The extra Tiny Context gates exist because Superpowers alone can still drift under long-running execution pressure: it improves execution discipline, but it does not by itself preserve source authority, prevent scope shrinkage, prove full conformance to the Technical Realization Plan or enforce AC-by-AC evidence against the Acceptance Checklist. Tiny Context gates may wrap Superpowers for authority, conformance and acceptance, but they must not redefine, duplicate or fork official Superpowers execution mechanics; future boundary conflicts must be surfaced instead of silently merged. In Codex Goal mode, implementation / execution goals use `ty-context composite-long-task final-gate` computed `product_goal_complete=true` as the active goal completion condition; agents must not hand-set it. A deliberately read-only audit / reporting goal may finish at `audit_task_complete` as a meta-goal, but must still state `acceptance_target_status`. Final wording distinguishes `audit_task_complete`, `acceptance_target_status` and `product_goal_complete`; if the verdict is not accepted/complete, the response must say `Audit workflow completed; acceptance target not complete.` and must not use unqualified `Goal achieved` or `update_goal(status="complete")` as product acceptance. The Superpowers-backed path treats acceptance as an external-reviewer evidence protocol: complete ACs need checklist-derived proof chains, fresh evidence, no missing required layers, no material drift, no unapproved sibling substitution and, when subagents are available, a read-only auditor pass that looks for gaps without becoming proof itself. Product / Architecture Source, Technical Realization Plan and Acceptance Checklist are the upstream authorities; state, audit, matrix, verdict, validator output, proof index and auditor report cannot rewrite them. `workflow-protocol.md` is a per-task execution protocol snapshot and must not be registered in `project_context/context.toml` or treated as business Context. `validate-superpowers-state` validates canonical state directly, while `validate-plan-acceptance` validates state-backed derived artifacts when `task-state.json` exists and preserves legacy matrix/verdict validation for old workdirs. Neither Skill nor validator executes the plan, proves product acceptance by itself, or creates durable Context.
- Composite long-task strict completion is current-attempt-only and owned by one Trusted Evidence Kernel used by final gate, `validate-superpowers-state`, state-backed `validate-plan-acceptance` and derived completion views. The kernel recomputes from current source hashes, current attempt identity, worktree fingerprint, required command specs/runs, registered EvidenceRecordV2 records, under-specified AC checks, AC-010 bootstrap checks, Harness Drift Lock, protected baseline guard and current contradiction scan. `product_goal_complete=true` can only be produced by that current final-gate recomputation. Stale attempt evidence, historical `events.ndjson` completion events, old `derived/**` completion views, matrix/verdict/evidence-index/final-summary rows, validator pass, final cards, auditor prose, AC summary-only proof, unregistered temporary JSON and hand-written status files are never proof. AC-010 summary-only bootstrap prevention is a kernel rule, not a derived-view warning. Current failures from command runs, Playwright/JUnit/test-result artifacts, DOM/owner-surface forbidden states, task-state false/partial status or derived/state mismatch block machine-backed completion for the affected AC/layer. HFC-001 is a `harness_task`: it may change harness completion surfaces only with adversarial fixtures and a happy-path fixture, and it must not claim product completion. HFC-002 extends this authority from proof computation to official generated completion output: `completion_output_status` is the only generated final-answer status and is limited to `accept`, `reject` or `blocked`; `accept` requires current final-gate recomputation with `product_goal_complete=true`, accepted acceptance target status, successful required validators and no persistent generated-output mismatch. Final summaries, final cards, matrix/verdict metadata, local audit, CLI output, goal objectives, execution bindings, workflow protocol snapshots and state-backed validators must render or validate from that resolver instead of promoting validator pass, matrix/verdict rows, final cards, audit completion or stale prose into completion. `audit_task_complete` remains separate from product completion. HFC-002 is also a `harness_task`: it may add output-gate fixtures and managed wording, but it must not rewrite the HFC-001 Trusted Evidence Kernel or expand into the full HFC-003 false-completion regression suite. HFC-003 is the durable false-completion regression suite: it is a package-owned `harness_task` fixture manifest plus runner that keeps HFC-001/HFC-002 non-regression coverage, blocks known stale/current-attempt, unregistered evidence, summary-only, derived-output, validator-only, harness-drift, scope-leakage, UI-proof and semantic-negative-proof false-completion paths, and retains a happy path so the system is not made permanently false. Its required fixture manifest is a hard gate and includes the boundary semantic proof regressions `missing_boundary_semantic_proof_refresh_catalog` and `missing_boundary_semantic_proof_import_evidence`; HFC-003 must fail if any required fixture id, mini workdir or expected-outcome file is absent. HFC-003 proves only that official harness completion surfaces do not regress in known false-completion scenarios; it never proves a business/product task complete. HFC-004 extends HFC-001/002/003 without loosening them: final gate first snapshots previous final/gates/meta transient bookkeeping as audit context, computes a current `FinalGateCandidateState` from the Trusted Evidence Kernel, resolves candidate completion output, derives current views, scans generated output in a mode selected by the current candidate rather than old state, classifies blockers through `FinalGateBlockerTriage`, performs at most one self-recovery pass for transient bookkeeping or regenerable generated-output mismatch, then writes current final state, events and CLI output. Old `completion_output_status=blocked`, old `generated_output_mismatch=true` and stale final/gates/meta findings cannot drag a current kernel accept back to blocked; current user-visible false completion claims, missing evidence, stale/contradictory evidence, environment blockers, contract blockers and Harness Drift Lock blockers remain blocking with category and next action. Product-task attempts that change tests, assertion generators, evidence writers, final-gate/validator/derive/state reducers, composite workflow protocol/Skill assets or protected harness baseline paths are blocked by Harness Drift Lock; a product task cannot modify those harness surfaces and use the same run's result to prove product completion.
- Capability-first delivery scope is now part of the Superpowers three-input contract without adding a fourth input. Product / Architecture Source declares whether the upstream work is `system_capability_build`, `representative_sample_validation`, `full_population_operation` or `mixed_scope_requires_boundary`, whether full population is required, which representative samples do and do not validate the claim, and what backlog is explicitly out of scope. Technical Realization Plan items declare their own delivery scope, capability target, representative samples, full-population boundary and non-required population. Acceptance Checklist items declare acceptance scope, what each AC validates and does not validate, sample boundary and whether full population is required. Validators and generated views must keep reusable capability build, representative samples, real-object coverage and full-population operation separate: running several concrete objects does not by itself prove reusable automation is complete; implementing the framework does not prove every real object completed; and sample provider/interface/page evidence cannot substitute for all-provider/all-interface/all-platform or full-population completion unless the AC explicitly allows it. A source/plan/checklist disagreement between capability build and full-population operation is `scope_conflict_requires_decision` and blocks completion until resolved.
- Composite long-task input parsing is intentionally strict so references cannot overwrite definitions. Product / Architecture Source uses document-level fixed fields. Technical Realization Plan PI definitions must be Markdown headings such as `## PI-001: ...`; Acceptance Checklist AC definitions must be headings such as `## AC-001: ...`. Inside each heading section, compile reads only fixed `key: value`, indented-list or `key: |` fields until the next same-or-higher heading. Plain references in prose, lists, tables, mapping previews and `related_*` fields are not definitions; legacy list-style PI/AC definitions with delivery fields now fail during `ty-context composite-long-task compile` with source file and line guidance.
- Composite long-task execution uses state-backed Slice Gate / Epoch Gate / Final Gate cadence to keep per-slice progress light while preserving strict final acceptance. Progress Accounting separates AC acceptance completion, engineering implementation progress, runtime/proof progress and workflow overhead in state and generated views. Multi-slice runs maintain proof-layer milestone statuses and Next 3-5 high-value clusters under `tmp/ty-context/plan-acceptance/**`; workflow overhead backpressure batches shared provider/browser/runtime/security epoch proof environments instead of repeatedly running the full final gate.
- Product Surface Contract workflow is the Minimal Context path for turning broad page/UI/product positioning principles into concrete project-owned surface responsibilities. The package provides a business-agnostic `context_surface_contract` Skill and a compact `product-surface-contract.md` template; user-project facts live in `project_context/**` using existing roles such as `contract`, `area`, `subdomain`, `verification`, `decision-rationale` and `implementation-index`. The Harness must not add a new surface-specific context role or create business surface contracts during `init`/`upgrade`; plan validators may only check declared temporary surface bindings for structural consistency, not product surface quality.
- AGENTS placement policy is part of the Minimal Context design: `AGENTS.md` is a startup router and hard-boundary surface. For package consumers, long design reasoning defaults to compact `project_context/**` facts unless a project already has a local spec/design convention; in this source workspace, `PROJECT_SPEC.md` is the Harness workflow design-spec surface. Role procedures belong in Skills, human usage docs in README, and machine checks in validators/tests only when they fit the product boundary.
- Architecture/rationale support stays inside Task Contract compilation as lightweight explicit judgment: `Architecture Context Hit` and `Decision Rationale Hit` help high-risk tasks avoid missing controlling Context or durable design reasons, while `Context Delta: none|required` remains the only long-term fact decision point. This reduces missed architecture/rationale reads without restoring ADR, tech-plan or implementation-doc chains.
- Context updates for those changes should be as small as the fact allows, but not line-count limited: write enough durable context to guide implementation before code alignment. When a product, architecture, technical or acceptance source packet introduces durable constraints, the task-local `plan.md` or equivalent plan surface should include Source-to-Context Coverage so every durable source item is either covered by existing Context, updated into owning Context, task-local only, explicitly out of scope, awaiting user decision or marked under-scoped. Source-to-Context Coverage does not contain implementation paths; high-risk implementation work uses Context-to-Implementation Binding after Context is known. Small code tasks are local implementation tasks where existing Context is sufficient and no durable product, architecture, API/schema, runtime/state, verification/deployment, security/redaction or surface-ownership fact changes; they stay code-first and must not create full plan/trace tables unless they discover durable Context changes or expand into high-risk work. Automation may warn about context-first drift but must not become an edit-order gate.
- `plan.md` is the workflow-contract instance for complex or high-risk work, not a durable document chain. It keeps Source-to-Context Coverage, Context-to-Implementation Binding, Task Contract, implementation steps and Conformance visible while work is active; long-lived facts discovered there must be extracted into `project_context/**` or `DESIGN.md`, unresolved `under_scoped` / `new_context_required` coverage prevents claiming full source alignment, and non-bound implementation rows prevent claiming full Context-to-code alignment.
- The context-first clarification preserves the original plan-before-implementation principle while keeping Minimal Context slim: removing stage ceremony did not remove Context authority over durable product intent, engineering boundaries or contracts.
- Area Context is the product/domain ownership boundary. Role context files are read-purpose slices owned by an area or, only when truly cross-domain, by the project root.
- Managed guidance now includes a soft role placement scan for `project_context/areas/**` authoring and migration: use `area` / `domain` for product ownership, `subdomain` for smaller owned product context, `contract` for API/schema/event/interface semantics, `foundation` for stable concepts or vocabulary, `verification` / `deployment` for repeat-execution paths, `implementation-index` for code navigation, `decision-rationale` for stable design reasons and `archive` for non-default historical or external material.
- Verification and deployment role Context are Minimal Context surfaces for critical repeat-execution paths: they record only durable preparation, shortest command/path, expected stage or signal, acceptable warnings and known dead ends that reduce future retest, CI, deploy, runtime bootstrap or environment-initialization rediscovery. They must not keep one-off logs, raw outputs, temporary JSON, CI artifacts, release ledgers, reports or secrets.

## Current State

- vNext implementation is Minimal Context Harness.
- `init` creates `project_context/context.toml` with one default `main` product/domain area at `project_context/areas/main.md` and a default verification role context at `project_context/areas/main/verification.md`; `upgrade` migrates legacy `project_context/areas/**/*.md` into `project_context/areas/**/*.md` and registers recoverable Context graph files in the manifest.
- v4 `validate-context` requires `project_context/context.toml`; older config versions should run `upgrade` before relying on the v4 gate.
- Ad hoc CLI docs and managed Makefile wrappers use the canonical package-qualified entry `npx --yes --package project-tiny-context-harness@latest ty-context`; no legacy CLI alias is retained.
- Current CLI commands guard unsupported future schema major versions before applying v4 assumptions; write commands fail before modifying files.
- `validate-context` validates the Context graph structure, role names, paths and field shapes; non-area roles are semantic labels rather than writing-template gates.
- `validate-code-modularity` enforces touched handwritten source line-count risk as a separate maintainability gate; `validate-harness` composes `validate-context` and `validate-code-modularity`. `validate-plan-contract` checks temporary workflow-contract plan surfaces for internal consistency, path-reference existence and declared Context-to-Implementation binding; `validate-plan-acceptance` checks legacy long-task matrix/verdict artifacts when no state exists and state-backed derived artifacts when `task-state.json` exists; `validate-superpowers-state` checks canonical Superpowers task state, source hashes, graph references, evidence/proof-layer consistency, assertion-backed evidence for machine-verifiable layers, negative evidence contradictions, stale evidence, sibling substitution, auditor blockers, derived drift and final completion rules. Validator `errors` remain exit-code/blocking; `warnings` and `hygiene` report non-blocking stale prose, milestone or optional-artifact cleanup. Plan/state validators do not prove product quality and are not included in `validate-harness`.
- Current composite long-task final completion rejects weak evidence for machine-verifiable ACs. A `complete` claim for UI/browser/API/runtime/worker/data/integration/security/test/all-provider/cleanup proof layers must have fresh reviewable evidence with passed assertion results, zero command/assertion exit codes, matching AC/layer targets, passed positive and negative assertions, no failed/stale negative evidence scan and no forbidden owner-surface state. Derived matrix/verdict views may display assertion status and blockers, but they remain generated views and cannot become proof.
- Managed guidance and default Context templates distinguish verification/deployment path facts from test reports and release ledgers: area files own product/domain facts, `verification` role files own repeatable validation paths, and optional `deployment` role files own repeatable deploy/runtime/bootstrap paths.
- Managed guidance documents the canonical loops `context -> implementation -> verification -> context drift check` and `implementation discovery -> context update if long-term fact changed -> implementation alignment -> verification`, plus `source packet -> Source-to-Context Coverage -> Context Delta / Task Contract -> Context-to-Implementation Binding -> implementation -> Contract Conformance` for high-risk source inputs.
- Old stage-based assets, state files and work-product trees are removed from the current source tree.
- Historical stage design is summarized in `PROJECT_SPEC.md`; legacy stage semantic migration support has been removed after user migrations completed.
- Delivery benchmark prompts should evaluate Minimal Context behavior for new Harness runs; old stage-based public result data has been removed.
