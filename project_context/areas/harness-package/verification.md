# Harness Package Verification

## Developer Feedback Tiers

Use the cheapest reliable feedback loop for the current change, then widen before release. These commands optimize development feedback only; they do not replace complete CI/release gates, the project tests declared by a Delivery Contract, or the Long-Task Final Gate.

- `npm run test:affected:list`: inspect the deterministic plan without building or running tests.
- `npm run test:affected`: discover changed paths, build at most once when required, and run mapped focused tests. In inferred local discovery only, untracked `.work_products/**` scratch is omitted and reported; tracked changes and explicit `--path` values are never omitted. Known Long-Task hot spots select named regressions; unmapped Long-Task runtime changes fail safe to the Trust Boundary Gate; shared fixtures, package/dependency or unknown changes fail safe to complete suites.
- `npm run test:long-task:focused`: one-build focused Long-Task regression loop for Context evolution, authority/progress, design boundaries and workflow entry points.
- `npm run test:delivery-contract:focused`: one-build focused Contract authoring/compiler/coverage/risk loop.
- `npm run test:affected -- --no-build`: reuse `dist` only after a deterministic build-input fingerprint proves it was produced from the current package source, TypeScript configuration, package metadata, lockfile and fingerprint owner. Missing or stale metadata fails before any test starts.
- Package CI runs a lightweight `windows-latest` real-subprocess probe for the affected-test npm launcher; Windows must route npm through `ComSpec` rather than spawning `npm.cmd` directly.

Affected and focused results are non-authoritative developer feedback. Before release or any claim that the package is fully validated, run the complete package, source-parity, smoke, pack and release gates below.

## Suite Purpose And ROI Design Index

Package verification has three executable tiers with distinct purposes:

- affected and focused commands are non-authoritative task-local defect localization;
- `npm run test:long-task:trust` is the bounded Long-Task Trust Boundary Gate for high-impact false-completion and cross-module regressions after the candidate diff is frozen;
- `npm test` is the complete default plus Long-Task release regression for one final candidate.

`affected-test-plan-v2` reports `developer-feedback`, `trust-boundary` or `release-regression` plus its discovery source and reason. Stable discovery sources include `explicit-paths`, `local-worktree`, `local-head-parent`, explicit/environment bases and CI base SHA/ref forms. Explicit paths bypass inference; dirty local discovery uses only the current worktree; clean local discovery uses `HEAD^`; explicit/CI bases are exact and fail closed when missing. Inferred local discovery reports `ignored_untracked_local_artifacts` when it omits `.work_products/**`; comparison-tracked and explicit paths remain visible. An unmapped Long-Task runtime change widens to the Trust Boundary Gate while retaining affected tests not already contained by that gate; shared fixtures, package/dependency boundaries and unknown changes continue to widen to complete suites.

Pull-request Package CI builds once and runs the complete default suite plus the Trust Boundary Gate through package `test:trust:built`. `main` and release preparation retain `npm test`. Trusted Publishing runs that complete gate exactly once in its prepare job, then the protected publish job reuses the same uploaded tarball and runtime attestation without rebuilding, retesting or repacking. Every package-suite runner emits ephemeral `test-suite-timing-v2` diagnostics with one terminal record per selected file. Controlled Ubuntu CI supplies wide, suite-specific catastrophic-regression ceilings; exceeding one fails that CI command without changing test discovery or acceptance meaning. Timing output is never Context, workflow state or acceptance authority.

The canonical affected plan executes exactly one highest selected aggregate. A complete Long-Task or release-regression selection contains the Trust files and therefore supersedes a separate Trust invocation; no persistent run history is needed or allowed to enforce this inside the canonical command.

The complete suite is not an edit/fix loop. Run it zero times during ordinary repair and once after diff freeze when routing or release requires it. After an aggregate failure, tracked source/test/config/shared-runner changes, suspected cross-suite contamination, or a run that itself owns the final validation claim must regain one clean complete pass on the newly frozen snapshot.

A proven local environment-only failure may stop at the failed test plus affected/Trust repair evidence without repeating the local complete suite only when tracked verification inputs are unchanged and a guaranteed downstream `main`/release gate will run the complete suite. The local aggregate remains failed and must never be reported as fully validated; without that downstream gate, a clean complete rerun is still required. A third run requires an explicit flaky, infrastructure or late release-mutation reason.

`dry_run: true` is an optional prepare-only diagnostic, not a mandatory rehearsal before a real publish. A real workflow dispatch prepares, tests, packs and smokes once; after the `npm-publish` environment approval, publication verifies the same workflow artifact against the dispatch commit, stable lockfile identity and tarball SHA-256. Node/npm versions are recorded as build provenance but the publisher job need not reproduce them because it never rebuilds. Lockfile identity normalizes CRLF/LF only; semantic or other byte changes still fail closed.

Sub-second status/resume budgets belong only to `test:long-task-performance`. Functional release-regression tests assert behavior without machine-load-sensitive local or cross-environment wall-clock thresholds. The only aggregate wall-time guard is opt-in, environment-specific and generously above the measured Ubuntu path; local runs remain diagnostic.

Canonical policy also sets reviewed maximums for Trust files, focused-tier files and one hotspot's mapped fan-out. An over-budget edit fails at policy load until the reviewer deliberately updates the named budget and rationale. These limits prevent silent feedback-tier cost growth; they never limit complete-suite discovery or justify deleting an independent invariant sentinel.

Long-Task fixture initialization may prepare default and external-confirmation Git seeds once per suite process, then copy the selected immutable seed into each unique temporary fixture. Every copy retains its own `.git`, common directory, worktree, local config and refs, has no remote and cannot observe another fixture's mutations. Direct single-file execution lazily creates the same seed semantics and removes them at process exit.

File-level concurrency uses explicit reviewed classes only: pure/static and proven isolated temporary-repository files enter the bounded safe lane at the reviewed default of two; Hook/profile, environment, distribution, verifier-migration, Git-race and unclassified files use the serial exclusive lane. Lane membership is disjoint and exhaustive, a new file defaults to exclusive, and `TY_CONTEXT_LONG_TASK_ISOLATED_CONCURRENCY=1` is the mechanical serial rollback. The default was enabled only after one serial and two concurrency-two runs over six representative lifecycle files preserved all 33 test identities, terminal outcomes, immutable seed/workspace state and cleanup while reducing the representative wall time from 206.276 seconds to 147.228 and 128.054 seconds; this bounded Windows observation is rollout evidence, not a cross-environment benchmark.

The implementation, migration status, measurement rules and stable retrieval keys are indexed in `docs/test-suite-roi-redesign.md` under `TS-PURPOSE`, `TS-BASELINE`, `TS-TIER-DEV`, `TS-TIER-TRUST`, `TS-TIER-RELEASE`, `TS-ROUTING`, `TS-RERUN`, `TS-RELEASE-HANDOFF`, `TS-OPTIMIZE`, `TS-MIGRATION`, `TS-METRICS`, `TS-AC` and `TS-NONGOALS`.

## Controlled Real CLI Smoke

A real temporary Git repository must prove:

1. initialize and compile one two-Outcome Delivery Contract;
2. first Compile emits the one-time execution-model checkpoint and an unchanged later Compile does not repeat it;
3. first Outcome targeted verify passes and second fails;
4. two self-consistent forged Receipts and a recomputed cache cannot accept; Stop executes both Checks and rejects;
5. after implementation repair, both targeted verify and one-snapshot Final Gate pass;
6. source code drift makes Stop reject;
7. restoring/committing code lets Stop or close run the Live Gate and clear binding;
8. a graph-derived Supporting Context edit can revise without user approval and preserve unrelated fresh Progress, while a Controlling Context edit remains approval-bound;
9. no Campaign/SFC/Packet/Codex worker/extra worktree/branch/model retry, automatic model switch, persistent model route/checkpoint state or proactive parallel-subagent scheduler was created.

## Evidence Rules

- Tests/Context validators prove only their named property. They do not prove product completeness.
- Guidance/parity tests prove that architecture obligations are distributed and indexed; they do not prove that an individual Agent chose the best architecture. Repository-native checks prove only their declared observable invariants.
- Affected/focused developer loops are selection aids only and do not replace complete CI/release gates.
- Targeted verify is repair evidence only. Acceptance comes only from the currently executing source-recompiled Live Final Gate; no stored result is reusable authority.
- `progress_passing` may be reported only as targeted repair evidence, `progress_stale` is not a current pass, and `final_workflow_status: null` remains unfinished; tests protect this wording without adding a machine state.
- Preflight proves only static authoring readiness and is intentionally non-authoritative; a successful Preflight creates no reusable receipt or completion evidence.
- Store no one-off logs, reports, raw evidence, secrets or release ledgers in Context.
