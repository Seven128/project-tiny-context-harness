# Composite Long-Task Workflow V3

This package-managed Skill is generated by `ty-context sync`. Consumer projects must not edit it directly.

## Invocation boundary

Run only through explicit `/composite-long-task-workflow` invocation. Ordinary questions, bug fixes and long tasks do not start this workflow or create an active contract. Agent planning, decomposition, native subagents, implementation technique, TDD, tests and reviews remain implementation choices outside the workflow contract.

## Purpose

The workflow permits implementation drift while preventing drifted delivery. Completion meaning comes from the frozen three-input contract, current facts come from the verifier, `final-gate` recomputes acceptance from the current workspace, and the project-level Stop Hook permits completion only while that accepted result is fresh.

Every default step must prevent a named drift risk:

- Contract compilation prevents omitted Requirement/PI/Obligation/AC coverage and freezes the inputs, registered Context, graph, oracle/verifier identities and workdir.
- Intermediate verification finds behavior, scope and evidence drift and returns repair findings.
- Final full recomputation prevents stale, partial, historical or cross-workspace proof.
- The Stop Hook prevents a missing, non-accepted or stale final result from being reported as completion.

## Required inputs

The workdir must contain complete strict Contract V3 YAML authorities:

- `product-architecture-source.yaml` (`product-source-v3`)
- `technical-realization-plan.yaml` (`technical-plan-v3`)
- `acceptance-checklist.yaml` (`acceptance-checklist-v3`)

The compiled contract must preserve the complete Requirement → PI → atomic Obligation → observable Binding → AC → Proof → Spec → Counterfactual graph. Oracle V2 returns typed actual observations only; Harness-owned operators, population rules, binding projection and obligation-sensitive counterfactuals compute results without trusting Oracle pass/status fields. Product boundaries, non-completing outcomes and forbidden shortcuts require executable negative assertions. Full-population claims require a frozen enumerator and coverage rule. Manual-only, summary-only and verifier-free ACs are invalid.

The pre-stable executor supports one exact Node Oracle step, no network, and no environment refs, requirements or probes. Browser/package/project commands, network access and environment-dependent contracts fail compilation explicitly; they are not silently ignored or retained as half-enabled modes.

Do not infer, repair or weaken missing contract meaning. Return compiler diagnostics when a required product decision is absent. A genuine user decision is handled through ordinary task communication; it does not create `accepted` and is not a separate successful terminal state.

## One fixed loop

```text
ty-context composite-long-task init <workdir>
author the three strict V3 YAML inputs
ty-context composite-long-task compile <workdir>
implement freely
ty-context composite-long-task verify <workdir>
while current-status is needs_work: repair findings and verify again
ty-context composite-long-task final-gate <workdir>
while final result is needs_work: repair, verify and run final-gate again
accepted and still fresh: report completion
```

While a task is active, compiling the identical contract for the identical workdir is idempotent. Any changed Product source, Plan, acceptance input, registered Context, oracle/verifier identity, graph or workdir is rejected with `active_contract_changed`; it cannot silently replace or weaken the active contract.

`verify` executes frozen verification definitions, collects only verifier-owned fresh observations/artifacts and writes repair-oriented `current-status.json`. The agent cannot provide pass results, evidence records, assertion results or Requirement/PI/Obligation/AC status. Intermediate results never authorize acceptance.

`final-gate` rechecks the frozen input, Context and oracle/verifier hashes; runs every in-scope AC against the current workspace; recomputes Obligation, PI and Requirement results bottom-up; binds the result to the current workspace hash; and writes `final-result.json`. Historical passes are never final evidence.

## Completion and Stop

- `accepted` is the only successful terminal result. It means the latest complete final recomputation passed and the current workspace still matches it.
- `needs_work` is internal loop state. It always returns to implementation and is never a legal completion reply.
- A missing product/user decision may pause ordinary task work and be communicated to the user, but it does not authorize completion and must not be represented as `accepted`.

The Stop Hook is a no-op when no active task exists. With an active task it blocks a missing final result, `needs_work`, any non-accepted result and any workspace/final-result identity mismatch. It allows completion only while the latest `accepted` result remains fresh.

Recovery reads the active `compiled-contract.json`, `current-status.json` and `final-result.json`. `goal-objective.txt` is a thin recovery pointer, not an authority. Runtime results remain temporary and never enter `project_context/**`.

## Pre-stable threat boundary

Current guarantees cover declared Requirement/PI/Obligation/AC omissions, changed compile inputs or observers, stale or missing final results, post-final workspace drift and silent active-contract replacement in the normal CLI/Hook flow.

Active contracts and final results may use ordinary user-level, project-level storage. This is not a hostile-Host security boundary. The workflow does not claim resistance to a malicious same-user or administrator deleting state or Hooks, Credential Manager/Registry attacks, system-level Hook bypass, deletion of all host state, or kernel/sandbox escape. Do not add Rust Host Helpers, administrator privileges, AppContainer/WFP/complex ACL policy, external audit runners or cross-platform release infrastructure to the default pre-stable path.

## Default verification budget

- Focused mechanism tests: at most 5 minutes total.
- Default Composite suite: at most 15 minutes total.
- One test: normally at most 2 minutes.
- Default tests must not install cross-platform VMs, containers, browser matrices or administrator environments.
- If a test exceeds budget, simplify it or move it out of the default path; do not hide the problem by only increasing a timeout.

The minimum real black-box matrix runs the CLI and `final-gate`, not source-regex or file-name substitutes:

1. `happy_path_real_implementation`
2. `missing_obligation`
3. `source_changed_after_compile`
4. `oracle_or_verifier_changed_after_compile`
5. `stale_or_missing_final_result`
6. `drift_repair_end_to_end`

Each case must name the concrete drift it prevents. Do not restore Superpowers, slice/epoch/task-state, derived reports, progress ledgers, forced TDD, forced subagent review or multiple execution modes.
